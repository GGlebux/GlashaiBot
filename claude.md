# Глашатай 📯 — гайд проекта

Телеграм-бот, который превращает голосовые сообщения и видео-кружки в текст
и делает их краткое содержание. Распознавание — локальный Whisper, саммари — GigaChat (Сбер).

## Что это

- **STT** (голос → текст): переключаемый через `STT_BACKEND`:
  - `whisper` — локальный **faster-whisper** (по умолчанию; бесплатно, без квот,
    модель зашита в образ). VPS 2 ГБ → `small` (дефолт); 1 ГБ → `tiny`/`base`.
  - `salutespeech` — облако **SaluteSpeech** (Сбер); требует подключённого пакета
    (иначе 402). Оставлен в коде как альтернатива.
- **LLM** (саммари): **GigaChat** (Сбер), облачный API.
- Сервисы Сбера ходят через OAuth2 и требуют корневые сертификаты
  **НУЦ Минцифры** (устанавливаются в Docker-образ).

> Историческая справка: изначально STT был на SaluteSpeech, но у физлиц free-пакет
> распознавания не подключался (ошибка 402), поэтому дефолтом сделали Whisper.

## Режимы работы бота

1. **Обычный** — пользователь присылает одно гс/кружок. Бот шлёт сообщение с
   **живым прогресс-баром** (`▰▱`, 0→100%), которое по стадиям обновляется
   (🎧 принял → 📥 скачиваю → ✍️ распознаю → 🧠 выжимка) и в конце превращается
   в **краткую выжимку тезисами** (с эмодзи) + строку **тон/стиль**. Полный текст
   приходит **отдельным сообщением ниже**. Реакции на исходном сообщении:
   `👀` принял → `🎉` готово (или `⚡` при ошибке).
2. **Цепочка** — `/begin` открывает сессию и шлёт одно статус-сообщение
   «🔗 принято: N», которое воркер **подбрасывает вниз** (удаляет+пересылает)
   после каждого гс, чтобы оно всегда было последним. Тексты копятся в Redis;
   `/end` → **одно общее краткое содержание** тезисами; `/cancel` отменяет.

> Реакции ограничены фиксированным набором Telegram (`👀 ✍️ 👍 🎉 ⚡`),
> см. `app/bot/reactions.py`. Прогресс-бар — `app/bot/progress.py`.

## Ключевые продуктовые решения

| Аспект | Решение |
|---|---|
| Доступ | Публичный + лимит `DAILY_LIMIT` сообщений/сутки на юзера |
| Чаты | Только личка (DM) |
| Админка | Вход через Telegram Login Widget, доступ по `ADMIN_IDS` |
| Хранение | **Только статистика/счётчики.** Контент (аудио, тексты, саммари) НЕ сохраняем |

Принцип приватности: расшифровки и саммари отправляются пользователю и нигде
не персистятся. В БД лежат только обезличенные метрики (счётчики, длительности,
статусы, тайминги).

## Архитектура

Один Docker-образ, три процесса (разные команды запуска) + Postgres + Redis + Caddy:

```
bot      (aiogram 3)   — приём сообщений, реакции, лимиты, постановка задач в очередь
worker   (arq)         — тяжёлая работа: ffmpeg → STT (Whisper) → саммари (GigaChat)
admin    (FastAPI)     — веб-дашборд: статистика, управление, здоровье бота
postgres               — метрики
redis                  — очередь arq + сессии режима /begin
caddy                  — reverse-proxy с авто-HTTPS (Let's Encrypt) перед админкой
```

Поток обычного режима: `bot` принимает гс → ставит `👀` и шлёт прогресс-бар →
кладёт задачу в arq → `worker` качает файл, прогоняет через ffmpeg → Whisper,
затем текст в GigaChat (`summarize_rich`) → редактирует прогресс-сообщение в
выжимку и шлёт полный текст отдельным сообщением.

## Структура кода

```
app/
  config.py            — настройки (pydantic-settings, читает .env)
  logging.py           — настройка логов
  db/
    base.py            — async engine/session
    models.py          — модели метрик (User, UsageEvent, DailyStat)
  services/
    stt.py             — диспетчер STT (выбор движка по STT_BACKEND)
    whisper_stt.py     — локальный STT (faster-whisper)
    sber_auth.py       — общий OAuth2-клиент Сбера (кэш токена на 30 мин)
    salute_speech.py   — облачный STT (SaluteSpeech), альтернатива
    gigachat.py        — клиент саммари
  bot/
    main.py            — entrypoint бота
    reactions.py       — реакции-статусы (👀 ✍️ 👍 🎉 ⚡)
    progress.py        — прогресс-бар и «подбрасывание» статус-сообщений
    middlewares.py     — только личка, апсерт юзера, проверка блокировки
    texts.py           — тексты/шаблоны сообщений (голос бренда «Глашатай»)
    handlers/          — common (/start /help), media (гс/кружки), chain (/begin /end)
  store.py             — Redis: суточные лимиты и сессии цепочки
  queue.py             — пул arq; redis_client.py — клиент Redis
  scripts/check_api.py — диагностика STT + GigaChat
  worker/
    main.py            — WorkerSettings; run.py — entrypoint
    tasks.py           — process_single / process_chain_item / process_chain_finalize
    audio.py           — конвертация в PCM 16к через ffmpeg
  admin/
    main.py            — FastAPI app
    auth.py            — проверка Telegram Login Widget
    routes.py          — дашборд и управление
    templates/         — Jinja2-шаблоны
```

## Запуск

```bash
# Прод (берёт готовый образ из Docker Hub)
docker compose up -d

# Разработка (собирает локально, монтирует исходники, hot-reload)
docker compose -f docker-compose-dev.yml up --build
```

Перед запуском: `cp .env.example .env` и заполнить токены (см. сам `.env.example`).

## CI/CD и авто-деплой

Схема: **GitHub Actions → Docker Hub → Watchtower на сервере**.

1. `.github/workflows/build.yml` на push в `main` (и по тегам `v*`) собирает
   единый образ и пушит в **Docker Hub** (`<user>/glashatai:latest` + тег по sha).
   Нужны секреты репозитория: `DOCKERHUB_USERNAME`, `DOCKERHUB_TOKEN`.
2. На удалённом сервере поднят `docker-compose.yml`, образ берётся из Docker Hub
   (переменная `DOCKER_IMAGE` в `.env`).
3. Сервис **Watchtower** (в составе compose) каждые 5 минут проверяет Docker Hub
   и при появлении нового образа **сам пере-создаёт контейнеры** — то есть деплой
   происходит автоматически по пушу в `main`, без ручного захода на сервер.

> TODO/на будущее: при желании заменить Watchtower на другой CI/CD-механизм
> доставки (например, SSH-деплой из Actions, webhook-триггер, или
> pull-агент). Сейчас выбран Watchtower как самый простой «poll-and-update».
> Для приватного репозитория Docker Hub задать Watchtower креды
> (`REPO_USER`/`REPO_PASS`, см. закомментированный блок в `docker-compose.yml`).

## Конвенции

- Язык интерфейса и комментариев — русский.
- Весь I/O асинхронный (aiogram, httpx, SQLAlchemy async, asyncpg).
- Секреты — только через `.env` (в репозиторий не коммитим).
- Контент пользователей не логируем и не сохраняем — только метрики.
- Зависимости с зафиксированными версиями в `requirements.txt`.
